# Web settings diag trim and local file dump

## Problem
- Web settings diagnostics became too heavy:
- Banner text was concatenating many runtime fields each refresh.
- Diag mode was effectively always on, with frequent polling.
- Large state JSON stayed in the page, causing UI lag.

## Changes
- `MFCMouseEffect/WebUI/app.js`
- Restored diag gating: diagnostics are enabled only when URL has `diag=1`.
- Reduced diag polling interval from `500ms` to `2500ms`.
- Reduced in-memory diag stream cap from `280` to `120` lines.
- Trimmed banner diagnostics to essential fields only (command totals + queue/encoder readiness).
- Removed state-dump UI actions that were adding unnecessary workflow.
- `MFCMouseEffect/MouseFx/Server/WebSettingsServer.cpp`
- Added automatic local dump in `BuildStateJson()` path (throttled).
- EXE writes current state JSON directly to:
- `.<exe_dir>/.local/diag/web_state_auto.json`
- No extra debug API endpoint is introduced.
- `MFCMouseEffect/WebUI/index.html`
- Removed state dump buttons and related textarea.
- `.gitignore`
- Added `/.local/` so dump files are not tracked.

## Why this fixes the lag
- Heavy diagnostics are no longer rendered in normal settings mode.
- Even in diag mode, refresh frequency and text payload are significantly reduced.
- Full state inspection is available in local file output generated by EXE directly.

## Manual verification
1. Open settings URL without `diag=1`: no diag panel and no high-frequency diag polling.
2. Open settings URL with `diag=1`: panel appears and updates are lighter.
3. Trigger normal settings refresh flow (open page/reload).
4. Confirm file exists at `.local/diag/web_state_auto.json` under exe directory.
